并发编程

上一节我们学了「异步」—— 一个人巧妙安排时间。这一节我们来学「并发」和「并行」—— 多个人一起干活。同样用生活例子和可交互动画帮你理解。

先看一个生活中的例子

🍳 场景：餐厅厨房要做三道菜
顺序 = 一个厨师做完一道再做下一道
并发 = 一个厨师「交替」做三道菜（炒一下这个，等它煮的时候去切那个）
并行 = 三个厨师同时各做一道菜

一句话总结： 顺序 = 一个人做完一件事再做下一件； 并发 = 一个人交替做多件事（看起来同时）； 并行 = 多个人真的同时做多件事。

并发 vs 并行 vs 异步

🔄
并发 (Concurrency)
交替执行多个任务，在一个 CPU 核心上快速切换，看起来「同时」。
🍳 一个厨师交替炒三道菜
⚡
并行 (Parallelism)
多个 CPU 核心真正同时执行多个任务。
🍳 三个厨师各炒各的菜
📨
异步 (Async)
发起请求后不等结果，先做别的事。上一节已学！
📱 下单外卖后继续做自己的事
概念说明Python 实现
并发交替执行多个任务多线程 threading
并行真正同时执行多进程 multiprocessing
异步非阻塞 I/Oasyncio（上一节已学）

注意：对于 CPU 密集型任务（数学计算等），Python 的多线程因为 GIL 的存在，速度并不会提升。 只有多进程才能利用多核实现真正并行。但对于 I/O 密集型任务（网络请求、读文件），多线程是有效的。

底层原理：GIL 到底是什么？

🔒 类比：会议室的麦克风
想象公司有一个会议室（CPU），三个人（线程）要开会。
GIL 就像只有一个麦克风 —— 虽然三个人都在会议室里，但同一时刻只有拿着麦克风的人能说话。
说了一段时间后，麦克风传给下一个人。所以虽然「三个人同时在会议室」（并发），但不是同时说话（非并行）。
要点说明
什么是 GIL同一时刻只允许一个线程执行 Python 字节码
影响CPU 密集型多线程无法利用多核
不影响I/O 密集型（网络、文件），GIL 在 I/O 等待时释放
解决CPU 密集型 → 多进程；I/O 密集型 → 多线程或异步
Django 主要是 I/O 密集型（数据库查询、HTTP 请求），多线程有效。

底层原理：线程与进程的内存模型

🏠 类比：合租 vs 独居
多线程 = 合租一套房子 —— 客厅、厨房是共享的，只有卧室是自己的。你改了客厅的灯，室友也能看到。
多进程 = 各住各的房子 —— 一切都是独立的。你在自己家怎么改，别人的家不受影响。

你在看数学书，突然要切换去看英语书。你需要：

为什么要了解上下文切换？
线程切换比进程切换快得多（因为线程共享内存，不用切换页表）。
但切换仍有开销 —— 创建太多线程反而会因为频繁切换导致性能下降，这就是为什么要用线程池来控制数量。

底层原理：竞态条件（最常见的并发 Bug）

🏧 类比：两人同时取同一个 ATM
账户余额 100 元。你和室友同时在两台 ATM 取 50 元：
✅ 正确：你取 50 → 余额变 50 → 室友取 50 → 余额变 0
❌ 竞态：两人同时读到余额 100 → 两人都取 50 → 余额变 50（少扣了 50！）
这就是竞态条件 —— 多个线程同时读写共享数据，结果不可预测。

第 0 / 6 步
如何避免竞态条件？
使用锁 (Lock)：with threading.Lock(): 保证同一时刻只有一个线程操作共享数据。
或使用线程安全的数据结构：如 queue.Queue。
Python 的 GIL 能避免一部分竞态，但不能完全避免（+= 这种复合操作仍不安全）。
用锁保护共享变量
import threading

counter = 0
lock = threading.Lock()

def safe_increment():
    global counter
    for _ in range(100000):
        with lock:            # 加锁：同一时刻只有一个线程能进入
            counter += 1      # 读取→计算→写回，三步不会被打断

threads = [threading.Thread(target=safe_increment) for _ in range(2)]
for t in threads: t.start()
for t in threads: t.join()
print(f"counter = {counter}")  # 永远是 200000 ✅

图解：线程池是怎么工作的？

🏊 类比：游泳馆的泳道
线程池 = 游泳馆。泳道数量（max_workers）是固定的。
当有人（任务）来游泳时，如果有空泳道就直接下水；如果泳道满了，就在外面排队等。
有人游完了，排队的第一个人立刻进入泳道。这样不需要反复「建泳道、拆泳道」，节省开销。

代码实战

多线程基本用法
import threading
import time

def download(name, seconds):
    print(f"  [{name}] 开始下载...")
    time.sleep(seconds)
    print(f"  [{name}] 完成 ({seconds}s)")

# 顺序执行
start = time.time()
download("文件A", 1)
download("文件B", 2)
print(f"顺序: {time.time() - start:.1f}s\n")

# 多线程并发
start = time.time()
t1 = threading.Thread(target=download, args=("文件A", 1))
t2 = threading.Thread(target=download, args=("文件B", 2))
t1.start(); t2.start()
t1.join(); t2.join()
print(f"多线程: {time.time() - start:.1f}s")
ThreadPoolExecutor
from concurrent.futures import ThreadPoolExecutor, as_completed
import time

def fetch_url(url, delay):
    time.sleep(delay)
    return f"{url} -> 200 OK ({delay}s)"

urls = [
    ("https://api.example.com/users", 1),
    ("https://api.example.com/posts", 2),
    ("https://api.example.com/comments", 1.5),
    ("https://api.example.com/albums", 0.8),
]

start = time.time()
with ThreadPoolExecutor(max_workers=4) as executor:
    futures = {executor.submit(fetch_url, url, d): url for url, d in urls}
    for future in as_completed(futures):
        print(f"  {future.result()}")

print(f"\n总耗时: {time.time() - start:.1f}s (顺序需 {sum(d for _, d in urls)}s)")

怎么选？一张图搞定

🌐
网络请求 / API 调用
大量 HTTP 请求、爬虫、调用第三方 API。等待网络响应是主要瓶颈。
asyncio / aiohttp threading
🗄️
数据库查询
Django ORM 查询、SQL 操作。等待数据库响应时 GIL 释放。
threading async ORM
🧮
数学计算 / 数据处理
图像处理、科学计算、大量循环。需要真正利用多个 CPU 核心。
multiprocessing
📁
文件读写
批量读写文件、日志处理。磁盘 I/O 等待时线程可切换。
threading
记住这个口诀：
I/O 密集 → 多线程 threading 或 异步 asyncio
CPU 密集 → 多进程 multiprocessing
Django 项目 → 大部分场景用多线程就够了！
