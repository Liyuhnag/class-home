1. 序列化与反序列化

在后端开发中，数据并不是一直以「对象」的形式存在。程序内部使用的是 Python 对象、类实例、列表、字典等内存结构，但当数据需要通过网络发送给前端、写入文件、保存到缓存、投递到消息队列时，就必须转换成另一种可以传输和存储的格式。这就是序列化与反序列化要解决的问题。

序列化，指的是把程序中的对象转换成可传输、可存储的数据格式。反序列化，指的是把这些格式重新转换成程序能够继续处理的数据对象。可以把它理解成一次「翻译」过程。程序内部的数据结构是机器运行时的表达方式，而 JSON、XML、YAML 这类格式更适合系统之间交换数据。

一、什么是序列化

序列化的本质是把内存中的对象转换成外部世界可以识别的形式。例如，一个 Python 字典本来只能在当前进程里使用，不能直接返回给浏览器，也不能直接放进消息队列中。只有把它转换成 JSON 字符串、二进制流或其他标准格式，别的程序或服务才能读取。

例如：

Python · json.dumps（序列化）
import json

data = {
    "name": "Tom",
    "age": 20
}

json_str = json.dumps(data)
print(json_str)

这里的 json.dumps(data) 就是在做序列化。它把 Python 字典转换成了 JSON 字符串。

二、什么是反序列化

反序列化是序列化的逆过程。它把已经传输或存储的数据格式，还原成程序可操作的对象。

例如：

Python · json.loads（反序列化）
import json

json_str = '{"name": "Tom", "age": 20}'
obj = json.loads(json_str)
print(obj)

这里的 json.loads(json_str) 就是在做反序列化。它把 JSON 字符串重新转成了 Python 字典。

三、为什么需要序列化与反序列化

序列化和反序列化之所以重要，是因为程序中的对象无法直接跨进程、跨网络或跨系统传递。

常见场景包括：

  • 前后端接口通信：后端把数据序列化成 JSON 返回给前端
  • 数据持久化：把对象转成可保存的格式写入文件、Redis 或数据库
  • 消息队列：把消息对象序列化后发送到队列中
  • 分布式系统：不同服务之间通过序列化后的数据进行调用和交换

如果没有序列化，不同系统之间就无法以统一格式传递数据。

四、常见的序列化格式

实际开发中常见的序列化格式主要有以下几类：

  • JSON：最常见，结构清晰，前后端都容易处理
  • XML：结构严谨，但相对冗长
  • YAML：可读性强，常用于配置文件
  • 二进制格式：例如 Protocol Buffers、MessagePack，体积更小，传输效率更高
  • 语言特定格式：例如 Python 的 pickle

在 Web 开发中，JSON 是最常用的格式，因为它跨语言、易调试、浏览器天然支持。

五、Django 中的序列化思想

在 Django 中，即使你没有显式使用「序列化器」这个词，也经常在做序列化工作。比如你在视图里返回 JSON：

Django · JsonResponse
from django.http import JsonResponse

def user_info(request):
    data = {
        "id": 1,
        "username": "admin",
        "email": "admin@example.com",
    }
    return JsonResponse(data)

这里 JsonResponse 做的事情，本质上就是把 Python 字典序列化成 JSON 响应。

前端接收到这段 JSON 后，再把它转成 JavaScript 对象进行展示和处理，这个过程本质上就是反序列化。

六、序列化与反序列化常见问题

虽然概念看起来简单，但在真实开发中有几个问题非常常见。

1. 类型不能直接转换

不是所有 Python 对象都能直接转成 JSON。例如日期、文件对象、自定义类实例，通常都需要先手动转换。

2. 外部输入需要校验

反序列化时不能只是「转回来」就结束，还需要校验字段是否合法。例如：

  • 用户名是否为空
  • 年龄是否是数字
  • 邮箱格式是否正确

否则系统可能接收到错误甚至恶意数据。

3. 安全问题

某些序列化方式虽然方便，但安全性差。例如 pickle 可以还原复杂对象，但如果反序列化不可信数据，可能带来严重安全风险。因此 Web 场景下通常优先选 JSON。

4. 版本兼容问题

如果一个系统升级后修改了序列化格式，旧数据可能无法正确反序列化。因此在大型项目中，数据格式设计通常要考虑兼容性。

七、如何理解这两个概念

可以用一句话总结：

  • 序列化：把程序内部的数据转换成外部可以传输和存储的格式
  • 反序列化：把外部格式还原成程序可以继续使用的数据结构

它们是接口开发、缓存、消息队列、前后端交互、分布式系统中的基础能力。

2. 序列化实践：serializer_auto_demo

下面结合 Django REST framework 演示项目，看「模型 → 序列化器 → JSON」在 API 里如何落地。概念上，这与前一节的序列化思想一致：把内存中的模型对象变成可传输的 JSON；DRF 的 Serializer 还承担字段校验与形状约定。

2.1 数据怎么流？

以 GET /api/books/ 为例：

  1. 视图用 Book.objects.all() 从数据库取出 QuerySet。
  2. BookSerializer(queryset, many=True) 把每条 Book 转成适合输出的基础类型（many=True 表示多条）。
  3. Response({ ..., "results": serializer.data }) 由 DRF 把 serializer.data 渲染成 JSON 响应。

2.2 项目里该看哪些文件？

文件 作用
bookstore/models.py Book 模型：书名、作者、价格、出版日期、是否推荐等。
bookstore/serializers.py BookSerializer：继承 ModelSerializer，在 Meta.fields 里列出对外字段；display_title、stock_label 用 SerializerMethodField，通过 get_* 方法生成计算字段（不必在数据库里多存一列）。
bookstore/views.py book_list、book_detail：取对象或 QuerySet，构造 BookSerializer，用 serializer.data 放进 Response。

2.4 小节

序列化器像是 API 的对外形状说明书：哪些字段出现、是否只读、是否要校验。视图尽量保持简单：查数据 → 序列化 → Response。使用 DRF 时，你通常不用在视图里手写 json.dumps，也不用自己逐个字段拼大字典；序列化器负责字段长什么样，视图负责取模型、调用序列化器、返回 Response。

3. 中间件：middleware_mixin_demo

3.1 中间件是什么？

中间件是挂在 Django 处理请求与响应整条链路上的组件。一个请求会依次经过多个中间件；视图执行完后，响应再依次经过这些中间件（具体经过哪些钩子，取决于你怎么写中间件类）。

常见用途：打日志、鉴权、安全头、统计耗时、限流等。

3.2 本项目用到的钩子（继承 MiddlewareMixin）

本项目中的中间件继承 django.utils.deprecation.MiddlewareMixin，并实现了：

方法 大致时机
process_request(request) 请求进来后，可做处理（本演示里会打印日志）。
process_view(request, view_func, view_args, view_kwargs) 已经匹配到要调用的视图，但视图函数还没执行。
process_response(request, response) 视图已经返回了 HttpResponse，可以再改响应（本演示里会打印日志）。

（Django 还提供 process_exception 等，本仓库未演示。）

3.3 多个中间件时，顺序怎么记？

在 settings.py 里，MIDDLEWARE 列表从上到下是中间件的注册顺序。对本演示中的这套 process_*：

  • 请求阶段：process_request、process_view 都是 先列表里的第一个，再第二个（自上而下）。
  • 响应阶段：process_response 是 先第二个，再第一个（自下而上），像「先套上的外层后脱」。

环境变量 MIDDLEWARE_DEMO_MODE 控制演示模式：

  • single：只启用 SingleTraceMiddleware。
  • multi（默认）：FirstTraceMiddleware 在前，SecondTraceMiddleware 在后。

访问 http://127.0.0.1:8000/demo/ 时，终端里应看到类似下面的输出（与你的 README 一致即可）。

单个中间件

终端输出
[单个中间件] process_request
[单个中间件] process_view
[视图函数] demo_view
[单个中间件] process_response

多个中间件

3.5 小节

中间件适合放很多视图都要用到的横切逻辑。顺序写错可能导致日志或鉴权顺序不符合预期。真实项目里还有框架自带的 Session、CSRF、安全相关中间件，自定义的一般按团队规范插在合适位置。

4. Celery 与耗时任务：celery_async_demo

4.1 这个演示项目在代码里演示什么？

task_demo/views.py 里有一个 sleep_test_view：在处理请求时执行 time.sleep(3)。这会让你明显感到接口变慢（大约 3 秒才返回）。它用来说明：

  • 在同步处理模型下，这段时间里 Web 服务器的一个 worker 一直被占用；
  • 并发多时，大量「慢请求」会明显影响能同时服务的请求数量。
说明：本仓库这一项目主要用上面的接口演示「阻塞」；完整接入 Celery（celery.py、Worker、Redis 等）可作为进阶练习，见下文「练习建议」。

4.2 Celery 里几个常用概念

名词 含义
Task（任务） 用装饰器注册的一个函数，可以被异步投递执行（例如 Django 里常用 @shared_task）。
Broker 消息队列中间件（常见如 Redis、RabbitMQ），用来存放「还没被执行的任务消息」。
Worker 独立进程，从 Broker 取任务并执行里面的代码。
调用方式 Web 视图里通常调用 某任务.delay(参数)，会尽快返回任务 id 之类信息，不等于等任务跑完才返回页面。

可以记一句话：Web 进程尽量快速响应；重活放到队列里由 Worker 慢慢做；Broker 负责在两者之间传递任务消息。

4.3 若以后在 Django 里接 Celery，一般会看到什么？

  • 项目里可能有 celery.py：创建 Celery 应用并从 Django 的 settings 读配置。
  • settings.py 里配置 CELERY_BROKER_URL 等。
  • 某个 tasks.py 里定义任务；视图里 import 后使用 .delay()。

具体安装与启动命令以你课程提供的文档为准。

6. 常见问题（自学时可对照）

  1. 序列化器里能用当前请求 request 吗？
    可以。视图传 context={'request': request} 时，在序列化器里可通过 self.context['request'] 使用（例如按登录用户决定返回哪些字段）。

  2. 为什么 process_response 是多中间件时倒序执行？
    与请求路径对称：先进入的中间件后参与响应处理，类似一层层「包洋葱」再一层层拆开。

  3. Celery 必须用 Redis 吗？
    不必须。Broker 也可以用 RabbitMQ 等；Redis 在学习与小型部署里很常见。
